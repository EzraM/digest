# Digest Event Format Standards

This document defines the canonical event format standards for the Digest application. All events must conform to these standards to ensure consistency across the application.

## Core Event Structure

Every event in Digest follows the `DigestEvent` type structure:

```typescript
interface DigestEvent {
  id?: number;              // Auto-generated by EventLogger
  timestamp: number;        // Auto-generated by EventLogger  
  sessionId: string;        // Auto-generated by EventLogger
  eventType: EventType;     // Required: Type of event
  data: any;               // Required: Event-specific data
  metadata: EventMetadata; // Required: Additional metadata
  display: EventDisplay;   // Required: UI display information
}
```

## Event Types

The following event types are supported:

- `user_prompt` - User input prompts
- `model_call` - AI model API calls
- `model_response` - AI model responses
- `block_operation` - Document block operations
- `response_parsing` - Response parsing operations
- `system_event` - General system events

## Required Properties

### EventMetadata
```typescript
interface EventMetadata {
  requestId?: string;      // Optional: Request tracking ID
  cost?: number;          // Optional: API cost information
  tokens?: number;        // Optional: Token count
  timing?: {              // Optional: Timing information
    startTime: number;
    endTime?: number;
    duration?: number;
  };
  source?: string;        // Optional: Event source identifier
  [key: string]: any;     // Additional metadata as needed
}
```

### EventDisplay
```typescript
interface EventDisplay {
  title: string;          // Required: Display title
  description?: string;   // Optional: Brief description
  icon?: string;         // Optional: Display icon (emoji)
  details?: DetailBlock[]; // Optional: Detailed breakdown
}
```

### DetailBlock
```typescript
type DetailBlock = 
  | { type: 'text'; label: string; content: string; }
  | { type: 'code'; label: string; content: string; language?: string; }
```

## Event-Specific Data Formats

### user_prompt
```typescript
{
  eventType: 'user_prompt',
  data: {
    prompt: string;        // Required: User input
    context?: string;      // Optional: Additional context
  }
}
```

### model_call
```typescript
{
  eventType: 'model_call',
  data: {
    systemPrompt: string;  // Required: System prompt
    userPrompt: string;    // Required: User prompt
    fullRequest: any;      // Required: Complete API request
  }
}
```

### model_response
```typescript
{
  eventType: 'model_response',
  data: {
    rawResponse: string;   // Required: Raw model response
    usage?: {              // Optional: Token usage info
      inputTokens: number;
      outputTokens: number;
    };
  }
}
```

### response_parsing
```typescript
{
  eventType: 'response_parsing',
  data: {
    rawResponse: string;       // Required: Original response
    parsedXml?: any;          // Optional: Parsed XML structure
    proposedOperations: any[]; // Required: Extracted operations
    parseSuccess: boolean;     // Required: Success status
    parseError?: string;       // Optional: Error message
  }
}
```

### block_operation
```typescript
{
  eventType: 'block_operation',
  data: {
    operations: any[];         // Required: Block operations
    source: 'user' | 'ai' | 'sync'; // Required: Operation source
    result: any;              // Required: Operation result
    documentState?: any;      // Optional: Document state
  }
}
```

### system_event
```typescript
{
  eventType: 'system_event',
  data: {
    action: string;           // Required: Action performed
    details: any;             // Required: Action details
  }
}
```

## Usage Guidelines

### 1. Use EventLogger Convenience Methods

Always prefer the convenience methods in `EventLogger` for creating events:

```typescript
const eventLogger = getEventLogger();

// ✅ Good - Uses convenience method with proper display formatting
eventLogger.logUserPrompt("Hello world", "user interaction");

// ❌ Bad - Manual event creation without display
eventLogger.logEvent({
  eventType: 'user_prompt',
  data: { prompt: "Hello world" },
  metadata: {}
  // Missing display property!
});
```

### 2. Event Validation

All events are automatically validated before logging. Invalid events will throw an error with a descriptive message.

### 3. Display Guidelines

- **title**: Should be concise and descriptive (e.g., "User Prompt", "Model Response")
- **description**: Brief summary of the event content or outcome
- **icon**: Use relevant emojis (💬 for prompts, 🤖 for AI, ⚙️ for system, etc.)
- **details**: Provide structured breakdown for debugging/analysis

### 4. Metadata Best Practices

- Include `source` to identify the component creating the event
- Add `requestId` for tracking related events
- Include timing information for performance analysis
- Store cost information for API usage tracking

## Migration from Legacy Formats

If you encounter legacy event formats in the codebase:

1. **Missing display property**: Update to use EventLogger convenience methods
2. **Different data structure**: Map to the canonical format defined above
3. **Custom event types**: Extend the EventType union if genuinely needed

## Common Anti-Patterns to Avoid

❌ **Creating events without display property**
❌ **Using custom event types without updating the type definition**
❌ **Inconsistent data structure for the same event type**
❌ **Missing required fields in event-specific data**
❌ **Direct database insertion without going through EventLogger**

## Validation Errors

The EventLogger will throw descriptive errors for invalid events:

- "Missing eventType"
- "Missing display property" 
- "Invalid eventType: custom_type. Must be one of: user_prompt, model_call, ..."
- "user_prompt events must have data.prompt"
- "Invalid detail type at index 0: must be 'text' or 'code'"

## Integration Points

Events are used by:

- **EventCard component** (`src/components/EventCard.tsx`) - Displays events in UI
- **DebugEventService** (`src/services/DebugEventService.ts`) - Forwards events to renderer
- **Database persistence** - Events are stored in SQLite for analysis
- **Real-time subscribers** - Components can listen to specific event types
